# 记忆管理页面 /api/memory/file 400 Bad Request

日期：2026-07-23
影响页面：http://127.0.0.1:5173/memory （记忆管理页 `MemoryPage`）

## 1. 用户现象

打开「记忆管理」页面，左侧记忆源列表加载后，右侧默认选中的 `data/groups/main/CLAUDE.md`
内容区一直空白 / 报错 toast「加载记忆文件失败」。浏览器控制台持续抛出：

```
GET http://127.0.0.1:5173/api/memory/file?path=..%2F.deepthink%2Fdata%2Fgroups%2Fmain%2FCLAUDE.md 400 (Bad Request)
Fetch 加载失败：GET "http://127.0.0.1:5173/api/memory/file?path=..%2F.deepthink%2Fdata%2Fgroups%2Fmain%2FCLAUDE.md"
```

（注：`content_script.js:5157 b.getContext is not a function` 来自浏览器扩展，与本_issue_无关。）

## 2. 问题描述

后端 `/api/memory/sources` 返回给前端的 `source.path` 形如
`../.deepthink/data/groups/main/CLAUDE.md`（含 `..` 路径段）。`MemoryPage` 把该 path
原样拼到 `/api/memory/file?path=...` 回查内容。后端 `normalizeRelativePath` 为防目录穿越，
禁止任何 `..` / `.` 路径段，于是返回 400 `Invalid memory path`，导致内容加载失败。

## 3. 根因

`src/routes/memory.ts` 中，记忆相对路径的基准目录用了 `process.cwd()`，而非数据目录：

- `listMemorySources`（生成路径）：
  `relativePath = path.relative(process.cwd(), absolutePath)`
- `resolveMemoryPath`（解析路径）：
  `const absolute = path.resolve(process.cwd(), relativePath)`

而 `src/config.ts` 中 `DATA_DIR` 默认 `~/.deepthink/data`，与源码仓库目录 `~/deepthink` 平级。
开发模式从仓库根 `~/deepthink` 启动后端（`npm run dev` → `tsx src/index.ts`），此时
`process.cwd() = /home/me/deepthink`，于是：

```
path.relative('/home/me/deepthink', '/home/me/.deepthink/data/groups/main/CLAUDE.md')
= '../.deepthink/data/groups/main/CLAUDE.md'   ← 含 '..'，被 normalizeRelativePath 拒绝
```

随后 `classifyMemorySource` 期望路径前缀为 `data/...`，dev 下前缀变成 `..`，分类也会落到
fallback（次要副作用）。

**证据链**：

1. 报错 URL 本身即 `path=..%2F.deepthink%2Fdata%2Fgroups%2Fmain%2FCLAUDE.md` —— 说明
   `/api/memory/sources` 吐出了带 `..` 的路径。
2. 运行中的后端进程 cwd 实测：
   ```
   pid 630409 (tsx src/index.ts) cwd = /home/me/deepthink   # 仓库根，非数据目录
   pid 630409 environ: 未设置 DEEPTHINK_DATA_DIR             # 故 DATA_DIR=~/.deepthink/data
   ```
3. 用 tsx 复现基准换算（修复前）：
   ```
   MEMORY_BASE_DIR(=cwd) = /home/me/deepthink
   relative = ../.deepthink/data/groups/main/CLAUDE.md   # 含 '..'
   ```

设计缺陷：相对路径基准依赖「启动 cwd 恰好等于数据目录上层」这一隐含假设。一旦从仓库根
启动 dev 服务（最常见的开发姿势），假设即破裂。

## 4. 复现路径

1. 仓库 `~/deepthink` 下 `npm run dev`（后端 cwd = 仓库根）。
2. 浏览器登录 admin，进入 `/memory`。
3. 控制台见 `GET /api/memory/file?path=..%2F.deepthink%2F...` → 400。

## 5. 诊断方法

```bash
# 后端 cwd 是否在数据目录上层？
readlink /proc/$(pgrep -f "tsx src/index.ts" | head -1)/cwd

# 确认是否设置了自定义数据目录
cat /proc/$(pgrep -f "tsx src/index.ts" | head -1)/environ | tr '\0' '\n' | grep DEEPTHINK_DATA_DIR || echo "(未设置 → 用默认 ~/.deepthink/data)"

# 直接复现相对路径计算（仓库根 cwd 下）
node -e "console.log(require('path').relative('/home/me/deepthink','/home/me/.deepthink/data/groups/main/CLAUDE.md'))"
# => ../.deepthink/data/groups/main/CLAUDE.md
```

## 6. 修复方案

将记忆相对路径的基准从 `process.cwd()` 改为 `path.dirname(DATA_DIR)`（默认即 `~/.deepthink`），
与启动 cwd 解耦。改动集中在 `src/routes/memory.ts`，3 处：

```diff
 const MEMORY_DATA_DIR = path.join(DATA_DIR, 'memory');
+// 记忆路径以 DATA_DIR 的父目录为基准（默认 ~/.deepthink），使相对路径形如
+// `data/groups/...` / `data/memory/...`，与服务启动时的 process.cwd() 解耦，
+// 避免 cwd 不在数据目录上层时生成 `../...` 路径被安全校验拒绝（400）。
+const MEMORY_BASE_DIR = path.dirname(DATA_DIR);
```

```diff
 function resolveMemoryPath(relativePath, user) {
-  const absolute = path.resolve(process.cwd(), relativePath);
+  const absolute = path.resolve(MEMORY_BASE_DIR, relativePath);
```

```diff
 function listMemorySources(user) {
     ...
     const relativePath = path
-      .relative(process.cwd(), absolutePath)
+      .relative(MEMORY_BASE_DIR, absolutePath)
       .replace(/\\/g, '/');
```

**选型理由**：
- 基准取 `dirname(DATA_DIR)` 而非 `DATA_DIR` 本身，是为了保持路径前缀 `data/...`，与
  `classifyMemorySource` / `isBlockedMemoryPath` 既有的 `parts[0]==='data'` 假设一致，零回归。
- 不放宽 `normalizeRelativePath` 对 `..` 的拒绝——目录穿越防护保持原样，仍返回 400。
- 生产模式（cwd=`~/.deepthink`）下 `dirname(DATA_DIR)=~/.deepthink=cwd`，行为与改动前完全一致；
  仅 dev 模式（cwd=仓库根）由坏变好。

**验证**（`tests/memory-route-paths.test.ts`，挂载真实 memory 路由 + mock admin）：
- `GET /api/memory/file?path=data/groups/main/CLAUDE.md` → **200**（修复前 400）✅
- `GET /api/memory/file?path=../.deepthink/data/groups/main/CLAUDE.md` → **400**（穿越仍拒绝）✅
- tsx 复算：`path.relative(MEMORY_BASE_DIR, mainClaude) = data/groups/main/CLAUDE.md`，
  无 `..`，与 `resolveMemoryPath` 来回解析一致 ✅
- `npx tsc --noEmit` 通过 ✅

## 7. 处理卡住的状态

不适用。仅前端读取 400，无运行态卡死。

## 8. 经验沉淀 / 预防

- 凡是「后端返回相对路径给前端、前端再原样回查」的接口，路径基准必须锚定到一个稳定的
  配置目录（`DATA_DIR` 体系），而非 `process.cwd()`——后者随启动姿势变化。
- 巡检建议：在 `tests/` 增加一条「dev cwd ≠ 数据目录上层」场景的契约测试，断言
  `/api/memory/sources` 返回的 path 不含 `..` 段（本_issue_已随附 `memory-route-paths.test.ts`）。
- 注意：`better-sqlite3` 原生模块若与运行 node 的 ABI 不一致，`/api/memory/sources` 会 500；
  这是测试环境问题，非代码缺陷——部署/CI 需保证 `npm rebuild better-sqlite3` 与运行 node 匹配。
